[KDW> ] It is true that there are three different scripts at the PP that posts data to the PP database and then back to the DCM system, but only ONE that SYNCFORCE uses to ADD / UPDATE / DELETE / VERIFY from the PP which is called Account_Admin.asp.  The other two, send only a subset of the data, since there is no need to send all of the data and expose those values which would not change by the update and is used for the initial REGISTRATION and User initiated PROFILE UPDATE.
 
Each script posts only the KEY=VALUE for those fields that are identified as YES (required) or OPTIONAL in each of the ACTIONS as defined in the EURODCM_to_PP_06-25-2002. XLS spec, however those marked NO, not marked or marked as Internal, would only be posted to the PP Database and possibly back to the DCM system if the KEY=Value was supplied.  But these fields ARE NOT TO BE ACTED UPON by the EuroDCM receiving script.  This specification has not changed since the inception of the specification.
 
I have no clue what scripts Syncforce wrote, so I am at a disadvantage discerning whether or note their code is efficient.  What may have been overlooked by SyncForce was to build a generic handler for only the YES (required) or OPTIONAL fields independent of the action as opposed to one for each that requires the management of all fields whether they are appropriate for the action or not.  There is no sense in posting data for fields that have absolutely no bearing on the action.  An example is the Registration Form.  Only the Key=VALUE pairs sent to the EuroDCM system where (under User Register* column), that KEY field is marked with a YES (Required Field), or if an KEY marked OPTIONAL and has data otherwise the value for the Key is set to and returned as KEY=NULL.

Below is an update (because a few new fields were added such as Initials, etc.) of the original example script that I provided to Syncforce (similar to response.asp) to show how to create a generic handling/debug script.  It assumes that only those fields YES or OPTIONAL (with data) fields with DATA or the value='' (reset field value to NULL) are acted upon, all other fields (not YES or OPTIONAL) with or without data are to be ignored and may not be echoed back since there is no action for those fields.
 
A YES field is required and will always have data will be echoed back.
An Option field with data will always be echoed back.
An Option field without data will be set to NULL and echoed back.
A field not marked as YES or OPTIONAL is not acted upon and not echoed back depending on the action.
 
This example works for ALL types of posts to the PP.  It decodes the KEY and Value and depending on the Value, Scripts that portion of the SQL statement appropriate for the ACTION for DB updating.  All you do is an UPDATE UserData Set ( SQL ) WHERE ID=Account_ID.  What could be simpler?  I spelled it out for them !!!
